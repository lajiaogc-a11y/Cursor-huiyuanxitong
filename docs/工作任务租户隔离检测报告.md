# 工作任务功能 - 租户数据隔离检测报告

## 一、检测结论

| 模块 | 是否按租户隔离 | 说明 |
|------|----------------|------|
| 任务表 (tasks, task_items, task_posters 等) | ✅ 是 | 表有 tenant_id，RLS 策略按租户过滤 |
| 维护设置 (TasksSettings) - 任务列表/关闭 | ✅ 是 | 显式传入 tenantId 查询 |
| 维护历史 (TasksHistory) | ✅ 是 | 显式传入 tenantId 查询 |
| 海报库 (TasksPosters) | ✅ 是 | 显式传入 tenantId 查询 |
| 任务看板 (TaskBoardContent) | ✅ 是 | getMyTaskItems 依赖 RLS，员工仅能看本租户任务 |
| **生成未交易名单 (generateCustomerList)** | ✅ 已修复 | 已增加 tenantId 参数并按租户过滤 members/orders |

## 二、详细分析

### 1. 任务相关表 - 已正确隔离

- **task_templates**、**tasks**、**task_posters**：含 `tenant_id` 字段，RLS 策略 `tenant_id = 当前员工.tenant_id` 或平台超级管理员
- **task_items**、**task_item_logs**：通过 task 关联，间接按租户隔离
- 创建/更新/删除操作均传入 `tenantId` 并校验

### 2. 生成未交易名单 - 存在漏洞

**问题**：`generateCustomerList` 未接收 `tenantId` 参数，直接查询 `members` 和 `orders`。

**影响**：
- **普通租户用户**：RLS 已限制 members/orders 为本租户，行为正确
- **平台超级管理员**：RLS 允许查看全部数据，生成名单会混合所有租户的会员，造成数据混乱

**调用链**：`TasksSettings.handleGenerate` → `generateCustomerList({ start_date, end_date })`，未传入 `tenantId`。

## 三、修复说明（已完成）

1. **taskService.ts**：`generateCustomerList` 增加 `tenantId` 参数；当传入时，先查询该租户员工 ID，再按 `creator_id`/`recorder_id` 过滤 members、按 `creator_id`/`sales_user_id` 过滤 orders。
2. **TasksSettings.tsx**：调用时传入 `tenantId`，并在无租户时提示「请先选择租户」。
