> **范围**：基于仓库静态扫描 + 已知架构（Vite/React 前端、Node/Express API、MySQL、部分 Supabase/Edge 遗留能力）。  
> **结论性质**：技术体检与差距分析，**非**渗透测试或生产环境实测。

## 三、按严重程度排序的问题列表

### 致命（P0 — 上线前强烈建议解决）

1. **生产数据库备份与恢复无仓库级/runbook 证明**（MySQL 为主场景下，RDS 快照/物理备份/恢复演练未文档化 → 数据丢失不可接受）。
2. **无 CI/CD 流水线**：构建、测试、部署依赖人工，回归不可重复。
3. **无通过门禁的 E2E**：登录、下单、权限、会员路径无法自动回归。

### 高（P1）

4. **前后端数据层双轨**：业务已大量走 Node API + MySQL，但前端仍存在 **Supabase 直连**（如部分 `ErrorReportsPanel`）与 **API 上报** 并存 → 环境不一致时功能静默失败。
5. **CSP 关闭**（`server/src/app.ts` 中 Helmet CSP 为 `false`）→ XSS 风险面大于默认。
6. **错误监控入口不统一**：全局 reporter 走 API，部分面板走 Supabase → 排障与权限模型分裂。
7. **自助找回密码**：管理员重置存在，**员工自助邮箱/短信找回**若业务需要则缺失或不完整。

### 中（P2）

8. **部分数据仍 `useEffect` 拉取**（如 `useMemberSpinQuota`）与 React Query 全局策略不一致 → 重复请求与闪动。
9. **大列表虚拟化**：核心表格未证明普遍使用虚拟滚动 → 大数据量卡顿风险。
10. **操作幂等**：依赖后端与 UI 防抖；需对「提交类」关键路径做清单化测试（重复点击、重复领取）。
11. **备份恢复演练**：无定期演练记录与 RTO/RPO 指标。

### 低（P3）

12. 无障碍（WCAG）未系统审计；部分已有 skip link。
13. PWA/离线：非必须；当前偏传统 SPA。
14. 成本与预算监控：无。

## 五、体验问题清单

1. **首屏与主题**：深色默认策略曾专门调整（见 `STABILITY-REFACTOR.md`）→ 仍存在「用户期望跟随系统」与产品默认的取舍问题。
2. **懒加载切换**：`Suspense` + 细进度条；极端网络下仍有 chunk 失败 → 依赖 reload 上限策略。
3. **状态闪动**：Auth 缓存与 loading watchdog → 已缓解，但 **非 Query 数据源** 仍可能「先空后有」。
4. **导航**：跨子域（会员域/员工域）必须整页跳转 → 属架构行为，需在 UX 上减少误跳。
5. **平板/手机布局**：`MainLayout` / `MobileLayout` 分支多，**需设备矩阵实测**（旋转、安全区、键盘遮挡）。

## 六、安全问题清单

1. **CSP 关闭** — 建议至少对生产启用合理 CSP 报告模式再收紧。
2. **Token 存储**：localStorage JWT 常见方案；需配合 **XSS 治理 + 短有效期 + 刷新策略**（按现状评估）。
3. **CSRF**：Bearer Token 模式风险低于 cookie 会话，但仍需避免敏感操作 GET、校验 Origin（CORS 已白名单）。
4. **错误信息**：避免向普通用户暴露堆栈与内部路径（ErrorBoundary 需审查）。
5. **权限绕过**：前端路由拦截不能替代后端鉴权；**每个新 API 需 checklist**。
6. **限流**：生产 **`TRUST_PROXY=1`**（已强调）。
7. **硬编码密钥**：`.env.example` 仅示例；**禁止**将真实密钥提交 Git（持续 grep 与 pre-commit）。

## 七、性能问题清单

1. **JS bundle**：路由懒加载已做；仍需 **bundle 分析**（`vite build --report` 或插件）控制大户。
2. **图片**：需检查是否普遍 WebP/合适尺寸（未全量扫资源）。
3. **重复请求**：React Query 默认已较优；**遗留 useEffect fetch** 为重复源。
4. **长列表**：订单/会员量大时 **虚拟列表** 为建议项。
5. **Web Vitals**：已上报；需在面板设 **预算与告警**。

## 八、结构问题清单

1. **Supabase 类型与 API 混用**：长期应 **收敛为单一 BFF（Node API）** 或明确边界（哪些表只读、哪些废弃）。
2. **脚本爆炸**：`package.json` 大量 `db:*` — 建议 **文档化「唯一推荐迁移路径」**，其余标 legacy。
3. **大页面组件**：`ReportManagement`、`OrderManagement` 等 — 建议按 **领域拆子模块**。
4. **全局事件**：`window.dispatchEvent` 多处 — 可逐步换为 **Query invalidate 或轻量 event bus** 并文档化。

## 十二、架构问题与重构方案

**问题**：**双数据平面（Supabase 客户端 + Node API + MySQL）** 导致配置复杂、监控分裂、部分功能在「纯 MySQL 部署」下不可用。

**重构方向（二选一为主）**：

- **A. BFF 一统**：所有读写经 Node API；Supabase 仅保留历史或逐步下线。  
- **B. 明确混合边界**：文档列出「仅 Supabase 的表/能力」+ 部署检查脚本，避免误配。

## 十三、渲染 / 闪屏类根因（归纳）

| 现象 | 可能根因 |
|------|-----------|
| 首屏深浅色跳变 | 主题初始化与 `localStorage` 读取时机（已部分用独立脚本缓解，见稳定性文档） |
| 路由切换闪一下 | 懒加载 `Suspense` fallback + 数据未命中缓存 |
| 数据先空后有 | 非 React Query 或 `enabled` 过晚、`staleTime` 与初始状态不一致 |
| Chunk 失败白屏 | CDN 缓存与旧 index 指向新 chunk — `main.tsx` 限制 reload 次数 |

## 十四、补丁叠加与清理方案

1. **标记 legacy**：`scripts/` 与 `db:*` 中仅历史迁移用的脚本移到 `scripts/legacy/` 或 README 表格说明「勿再执行」。  
2. **删除死代码**：配合 `knip`/`ts-prune` 做季度清理。  
3. **统一数据访问**：禁止新功能直接 `supabase.from`；统一 `api/client`。
