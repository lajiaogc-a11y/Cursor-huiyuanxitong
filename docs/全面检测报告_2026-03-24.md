## 3. 主要问题清单（按类别）

## 3.1 类型系统债务过高（TypeScript 退化）

**现象**

- `any` 泛滥，前后端均有，且集中在核心数据路径（如订单、租户、表代理、成员配置）。

**影响**

- 线上问题更容易在运行时才暴露。
- 重构风险极高，IDE 智能提示失真。
- 安全边界（字段过滤/租户隔离）更容易被不严格类型绕过。

**优先修复模块（建议先做）**

- `src/hooks/orders/orderQueries.ts`
- `src/services/tenantService.ts`
- `server/src/modules/admin/repository.ts`
- `server/src/modules/data/tableProxy.ts`
- `server/src/modules/data/repository.ts`
- `src/services/members/memberPortalSettingsService.ts`

**修复方案**

1. 新建类型基线策略：
   - 先禁"新增 any"（CI 里对增量做 gate），再逐步消化存量。
2. 在数据边界层引入运行时校验（Zod）+ 类型推导：
   - API 入参、DB 返回、RPC 返回都先 parse 再入业务层。
3. 对核心 DTO 建立 `types/` 统一定义：
   - `Tenant`, `Order`, `MemberPortalSettings`, `AuthSession`, `AuditRecord`。
4. 采用"模块清零"而非全仓一次性：
   - 每周 1~2 个模块去 any，避免冻结迭代。

## 3.2 构建告警显示存在潜在样式/打包问题

**现象**

- CSS 压缩阶段出现语法告警（`Expected identifier but found "-"`）。
- 多处模块出现"动态导入 + 静态导入并存"，导致拆包无效。

**影响**

- 样式异常在某些页面可能只在生产构建后触发。
- 分包策略形同虚设，首屏 JS 体积和缓存收益变差。

**修复方案**

1. 锁定 CSS 告警来源：
   - 用 source map 或分段禁用压缩定位具体源文件。
   - 修正非法 token（通常来自拼接字符串、异常内容注入、或转义问题）。
2. 统一"动态加载策略"：
   - 对高频公共依赖（如通知、数据服务）保留静态导入。
   - 对低频重页面完全动态导入，避免混用。
3. 加入 bundle 预算：
   - 设 chunk size budget，超过阈值 CI 警告。

## 3.3 规则冲突与代码异味（可快速治理）

**现象**

- `no-empty`、`prefer-const`、`no-require-imports`、`no-control-regex`、`no-useless-escape` 有持续性告警。

**影响**

- 空代码块会掩盖异常。
- `require`/ESM 混用增加运行时不一致概率。
- 正则表达式可读性和正确性下降。

**修复方案**

- 建立"一键修复+人工复核"流水线：`eslint --fix` + PR review checklist。
- 对 `no-empty` 必须改为显式注释并记录兜底行为。
- 将 require 动态分支改成标准 ESM `import()`。

## 3.4 测试结构不匹配业务复杂度

**现象**

- 单元测试均通过，但缺少关键场景的系统验证（特别是权限/多租户/数据迁移）。

**高风险缺口**

1. 角色权限矩阵回归（平台管理员/租户管理员/员工/会员）。
2. 租户隔离与越权写入回归（tableProxy + 关键路由）。
3. 关键业务链路 E2E：登录、下单、结算、积分、会员设置、任务海报、日志审计。
4. 失败重试与幂等：网络抖动、重复提交、超时回放。

**修复方案**

- 补"冒烟 E2E 套件"先保核心流程（10~20 条）。
- 后续按模块扩展到"鉴权失败/并发/异常路径"。
- 在 CI 加入 nightly 全链路回归。

## 7. 页面与功能维度"逐块"风险矩阵（静态审计视角）

基于路由配置与页面清单，当前站点至少覆盖：

- 员工端核心/扩展页（订单、会员、员工、汇率、商户、报表、系统设置、审计、日志、任务、知识库等）
- 会员端页（登录、首页、抽奖、积分、邀请、设置、落地页）

> 由于本次没有真实账号矩阵与线上数据，本节为"源码静态 + 构建告警 + 测试覆盖"视角，重点识别**高概率风险点**。

### 7.1 鉴权与路由层

- 风险点 A：路由数量多、角色层级复杂（`AdminProtectedRoute` + `ProtectedRoute` + `MemberProtectedRoute`），存在"前端可见但后端拒绝"或"前端隐藏但后端放行"的不一致风险。
- 风险点 B：旧路由重定向较多（Legacy redirect），查询参数和 hash 透传虽已处理，但缺少回归测试确保权限上下文不丢失。

修复建议：

1. 建立"页面×角色×接口"权限矩阵自动化测试。
2. 对所有 legacy 路由补 301/302 与鉴权一致性测试。

### 7.2 员工端业务模块

#### 订单/会员/员工/报表（高业务价值模块）

- 风险点：核心查询与聚合路径存在大量 `any`，影响金额、状态、筛选条件的类型安全。
- 风险点：大表场景依赖前端筛选与分页，缺少"超大数据量 + 并发筛选"压测基线。

修复建议：

1. 先治理订单与租户相关服务的类型边界（DTO + Zod parse）。
2. 增加关键报表一致性校验：页面汇总 = 导出结果 = API 聚合结果。

#### 系统设置/平台设置/成员门户设置

- 风险点：配置项极多，且页面复杂度高（此类页面 lint 告警密集），变更后极易出现配置联动回归。
- 风险点：动态导入与静态导入混用，设置页性能优化收益被削弱。

修复建议：

1. 按"配置域"拆分设置提交事务，避免一次提交影响多个子系统。
2. 为设置页建立冒烟回归脚本（保存/回滚/刷新后读取一致）。

#### 审计中心/操作日志/登录日志

- 风险点：日志链路依赖多模块上报，若空 catch 或弱类型存在，会出现"业务成功但审计缺失"。

修复建议：

1. 审计字段最小必填集（操作者、对象、动作、结果、时间、租户）做服务端强校验。
2. 对审计写入失败策略做显式定义（阻断/降级/告警）。

### 7.3 会员端模块

#### 会员登录、抽奖、积分、邀请

- 风险点：会员侧安全依赖 JWT 类型区分与限流器，当前能看到限流设计，但缺少"恶意重放/并发连点/越权 token"回归套件。
- 风险点：邀请落地/注册路径属于公开入口，是攻击面重点。

修复建议：

1. 增加会员业务风控用例：令牌篡改、过期 token、跨租户 token。
2. 抽奖与邀请接口补幂等键与审计关联 ID，便于追查。

### 7.4 任务系统与海报/手机号提取

- 风险点：涉及外部数据与导入导出，容易出现"字段兼容性 + 编码 + 空值策略"问题。

修复建议：

1. 为导入导出建立 schema 版本号，严格校验格式。
2. 提取/批处理操作增加 dry-run 预检与回滚包。

### 7.5 知识库与公共页面

- 风险点：公共可见数据接口若无缓存与限流策略，可能出现抓取放大。

修复建议：

1. 公共接口增加缓存层与严格频控。
2. 公共数据仅返回最小字段，避免内部字段暴露。

## 10. 你说的"改一处炸很多处"问题：根因与系统化优化方案

你描述的是典型的**高耦合 + 低边界清晰度**问题。结合当前仓库结构，主要根因通常有 5 类：

1. **共享状态过宽**：多个页面直接读写同一 store/service，缺少领域边界。
2. **类型约束不足**：`any` 使编译期无法拦截连锁影响。
3. **接口契约不稳定**：前后端字段命名、可选值、空值语义不一致。
4. **副作用分散**：一个动作同时触发 toast、日志、缓存刷新、列表重拉，且无统一编排。
5. **缺少防回归网**：改动后没有最小可执行回归矩阵，问题在上线后才暴露。

### 10.1 优化目标（先定义"好代码"标准）

把系统从"改一处影响全站"优化为"改一处只影响本域"，标准如下：

- 领域模块边界清晰（订单/会员/租户/设置/审计独立）
- 每个模块有稳定输入输出契约（DTO + schema）
- 副作用集中编排（统一 orchestrator）
- 变更有自动回归（单测 + 合同测试 + E2E 冒烟）

### 10.2 不改业务功能的重构策略（推荐顺序）

#### 第一步：做"依赖地图"与影响半径可视化（1 周）

目标：先看清"谁依赖谁"。

执行：

1. 对 `src/services`、`src/stores`、`src/hooks`、`server/src/modules` 生成依赖图。
2. 标出高扇出文件（被很多模块依赖）与环依赖。
3. 识别"改动高风险入口"清单（前 20 个文件）。

产出：

- 依赖热力图
- 高风险文件清单（强制 code owner review）

#### 第二步：建立"防扩散门禁"（当周即可）

目标：防止技术债继续增长。

执行：

1. CI 禁止新增 `any`（增量门禁，不追历史一次清零）。
2. 新增代码必须通过 schema 校验（Zod）后进入业务层。
3. 对关键目录开启更严格 lint 规则（先小范围试点）。

#### 第三步：按领域拆分 service/store（2~4 周）

目标：让变化局部化。

执行：

1. 把"跨域通用逻辑"抽成 domain service（只暴露稳定接口）。
2. 页面层禁止直接拼接后端字段，统一走 mapper/adapter。
3. store 只保留状态，不承载复杂业务流程；流程挪到 use-case 层。

实践原则：

- 一个 use-case 只处理一条业务意图（例如"提交订单并刷新相关视图"）。
- 副作用（toast/日志/缓存刷新）在 orchestrator 一处管理。

#### 第四步：接口契约治理（并行推进）

目标：消灭"字段变一下全站裂开"。

执行：

1. 前后端共享 DTO 定义（或至少共享 schema）。
2. 每个 API 都有 contract test（请求/响应样例固定）。
3. 对 breaking change 建立版本策略（v1/v2 或兼容层）。

#### 第五步：建立"改动即回归"的测试矩阵（持续）

目标：任何改动都有安全网。

执行：

1. 最小冒烟 E2E（20 条）：登录、权限、下单、结算、积分、抽奖、设置保存、审计查询。
2. 权限矩阵测试（角色 × 页面 × API）。
3. 多租户隔离测试（读写越权、跨租户 token）。

### 10.3 代码组织建议（你可以直接按这个做）

建议目标目录形态（示意）：

- `src/domains/orders/`（types, schema, service, useCases, selectors）
- `src/domains/members/`
- `src/domains/tenants/`
- `src/shared/api/`（统一请求、错误、重试、幂等键）
- `src/shared/state/`（最小全局状态）
- `src/features/<page>/`（页面只组装，不写核心业务）

关键规则：

1. `features` 只能依赖 `domains` 和 `shared`，不能跨 feature 直接互调。
2. `domains` 不依赖 UI。
3. `shared` 不包含业务规则。

### 10.4 你最该先动的 3 件事（ROI 最高）

1. **先把新增 any 禁掉**：立即降低"改一处崩多处"的概率。
2. **给订单/租户/设置做契约层（DTO+schema）**：先保护最容易连锁出错的核心路径。
3. **上 20 条核心 E2E 冒烟**：把"人工回归"改为"自动回归"。

### 10.5 你如何判断优化是否成功（量化指标）

- 变更失败率（Change Failure Rate）连续 4 周下降。
- 平均修复时间（MTTR）下降。
- PR 平均影响文件数下降（耦合降低）。
- 高风险模块回归缺陷数下降。
- lint 新增违规数维持 0。

## 11. 抽奖系统专项审计（前端逻辑 + 接口安全 + 业务一致性）

> 你这次重点是"抽奖系统前端逻辑漏洞 + 全方面检查 + 给一套完善逻辑方案（不直接修复）"。本节给出专项结论与开发执行方案。

### 11.1 已发现的高风险漏洞/逻辑问题

#### [高危] 管理端抽奖接口缺少角色鉴权

现状：

- `/api/lottery/admin/*` 仅挂了 `authMiddleware`，没有管理员角色校验。
- 理论上只要是已登录 token（含会员 token）就可能访问管理接口。

风险：

- 非管理员可读取或修改奖品、日志、设置。
- 若 `tenant_id` 为空路径被触发，可能影响"全局奖池/默认配置"。

#### [高危] 员工查询会员抽奖数据存在跨租户越权风险（IDOR）

现状：

- `quota/:memberId`、`logs/:memberId`、`prizes/:memberId` 对员工请求直接信任 URL 中的 memberId。
- 缺少"员工 tenant_id 与 member tenant_id 必须一致"的强校验。

风险：

- 员工若知道其他租户 memberId，可跨租户读取抽奖信息（严重数据隔离问题）。

#### [高危] 抽奖开关 `enabled` 未在抽奖主流程生效

现状：

- 抽奖设置保存了 `enabled`，但 `draw()` 主流程未读取并阻断关闭状态。

风险：

- 后台显示"已关闭抽奖"，前台仍可抽奖，形成业务与风控失效。

#### [中高] 前端展示奖池与后端真实抽奖奖池可能不一致

现状：

- 前端九宫格仅展示前 8 个启用奖品。
- 后端抽奖从全部启用奖品抽取。

风险：

- 可能出现"抽到一个页面没展示的奖品"，动画兜底会落在索引 0，造成用户对公平性的质疑。

#### [中] 抽奖次数模型语义可能不清晰（积分券是否可跨天累计）

现状：

- 剩余次数计算 = `daily_free + spin_credits总和 - 当日已抽次数`。
- 该模型等价于"积分券提高每日总上限"，不是"可累积并按次消耗"的券仓。

风险：

- 若产品预期是"券可累积、按次扣减"，当前实现将与预期不一致，引发投诉和账务争议。

### 11.2 中风险工程问题（稳定性/一致性）

1. 错误码较多但前后端 contract 未显式版本化，未来加错误码可能导致前端处理分支漂移。
2. 抽奖记录/次数/积分更新虽在事务内，但对外监控指标（成功率、重复请求率、超时率）未形成稳定看板。
3. 前端动画与真实结果绑定在客户端流程中，缺少"服务端结果签名/回放凭证"机制，审计可解释性不足。

### 11.3 抽奖系统"完善逻辑"目标模型（推荐）

#### A. 权限与租户隔离模型

- 管理端：必须 `employee + admin role`（建议细到 `lottery:read`/`lottery:write`）。
- 会员端：只能访问自己数据，URL 参数仅作冗余校验。
- 员工代查会员：必须校验 `employee.tenant_id === member.tenant_id`。

#### B. 抽奖状态机（统一业务语义）

建议把抽奖抽象为状态机：

1. `READY`（可抽）
2. `BLOCKED_DISABLED`（活动关闭）
3. `BLOCKED_QUOTA`（次数不足）
4. `BLOCKED_RISK`（风控限制）
5. `DRAWN`（已完成，返回结果与凭证）

前后端只传状态码，不传散乱文案，文案由前端 i18n 映射。

#### C. 次数账户模型（避免争议）

把可抽次数拆为两本账：

- `daily_free_quota`（按天刷新，不结转）
- `credit_quota_wallet`（可结转，按次扣减，保留流水）

消费顺序固定：先免费后券，且每次消费写 `quota_ledger`。

#### D. 奖池与展示一致性

- 服务端返回"展示奖池快照ID + 奖品列表 + 概率版本号"。
- 前端必须按该快照渲染；抽奖请求携带 `pool_version`。
- 若版本不一致，返回 `POOL_VERSION_MISMATCH`，前端先刷新奖池后再抽。

#### E. 可审计与防抵赖

每次抽奖返回：

- `draw_id`
- `pool_version`
- `entropy_ref`（随机源标识）
- `result_hash`
- `server_time`

用于客服申诉与风控复核。

### 11.4 你的开发执行方案（不改功能，仅治理）

#### 第 1 阶段（1 周，先堵漏洞）

1. 统一补齐 lottery 路由权限中间件（admin/member/tenant scope）。
2. 把跨租户 memberId 访问校验前置为通用 guard。
3. 在抽奖主流程接入 `enabled` 开关判定与错误码。

#### 第 2 阶段（1~2 周，修业务一致性）

1. 明确次数账户语义（产品评审 + 技术方案冻结）。
2. 奖池引入版本机制，前后端使用同一快照。
3. 整理错误码枚举，建立前后端共享 contract。

#### 第 3 阶段（2 周，强审计与防回归）

1. 为抽奖链路补 contract tests（含错误码分支）。
2. 上 E2E：并发点击、重复请求、跨角色、跨租户、开关关闭。
3. 建监控看板：draw success rate、duplicate rate、quota mismatch rate、p95 latency。

### 11.5 抽奖专项验收清单（上线前必须全绿）

1. 会员 token 无法访问 `/admin/*`。
2. 员工不能查询非本租户 member 抽奖数据。
3. 抽奖关闭时，所有端口都返回一致状态码。
4. 前端展示奖品与后端可抽奖池版本一致。
5. 抽奖结果可用 draw_id 在后台完整追溯。
6. 并发压测下无重复扣减/重复加分。

## 12. 会员前端体验专项（页面切换"闪跳"与整体体验）

> 你反馈"切换页面会闪跳，修复多次仍存在"。这一类问题通常不是单点 bug，而是路由结构、懒加载策略、滚动容器、壳层重挂载共同作用的结果。

### 12.1 已定位到的"闪跳"关键成因

#### 成因 1：会员布局不是"持久布局"，路由切页会重建壳层

现状：

- 会员每个受保护路由都包了一层 `MemberRoute`，而 `MemberRoute` 内部再包 `LazyMemberLayout`。
- 这意味着页面切换时，布局容器可能反复卸载/挂载（底栏、客服、滚动容器一起重建）。

影响：

- 视觉闪动、滚动位置重置、底栏轻微跳动。
- `html.member-html` class 在 mount/unmount 中反复切换，触发样式重算。

#### 成因 2：会员页面是懒加载 + 全局 Suspense fallback，切页会出现"空帧"

现状：

- 顶层使用单一 `<Suspense fallback={<TopProgressBar />}>`。
- 当目标页面 chunk 尚未就绪时，内容区可能短暂只剩进度条（或空），造成"闪一下"的感知。

影响：

- 尤其在弱网、首跳到某页时更明显。

#### 成因 3：会员端采用内部滚动容器，切路由时滚动上下文被重置

现状：

- `MemberLayout` 使用 `PullToRefresh` + 内部 `overflow-y-auto` 容器作为主滚动区。
- 路由变化时 `PullToRefresh` 会重置内部状态；壳层若重挂载，滚动位置会丢失。

影响：

- 用户感知为"跳顶""突然抖一下"。

#### 成因 4：下拉刷新采用 `window.location.reload()`，会引入整页重载闪烁

现状：

- 触发下拉刷新后是整页 reload，而不是数据级刷新。

影响：

- 一旦误触或频繁触发，体验接近"网页重新打开"。

#### 成因 5：会员页存在较多重内容页面，切换时布局高度变化大

现状：

- 抽奖页、商城页、设置页、首页结构差异大，且有异步加载。

影响：

- 在壳层不稳定时，高度变化被放大成"闪跳"。

### 12.2 会员体验度全量问题清单（除闪跳外）

1. 页面骨架与真实内容切换节奏不统一，部分页面先空后加载。
2. 多页并发请求缺少"路由切换时取消请求"规范，慢请求回写可能造成 UI 抖动。
3. 会员导航切换缺"过渡占位容器"，导致内容区高度突变。
4. 下拉刷新策略过重（整页 reload），对移动端体验不友好。
5. 缺少"会员端体验指标"看板：TTI、切页耗时、白屏时长、跳出率。

### 12.3 不改业务功能的修复开发方案（执行版）

#### 阶段 A（1 周）：先止闪跳

1. 路由重构为"持久会员布局"
   - 用父级 route + `Outlet` 固定 `MemberLayout`，子路由只替换内容区。
2. 对会员页面增加"局部 skeleton fallback"
   - 保持壳层和底栏常驻，避免 Suspense 导致整块空白。
3. 保留并恢复滚动位置
   - 按路由 key 记录/恢复内部滚动容器 scrollTop。

#### 阶段 B（1~2 周）：稳体验

1. 下拉刷新从"整页 reload"改为"数据刷新 orchestrator"。
2. 路由切换统一取消上页未完成请求（AbortController / query cancel）。
3. 为重页面配置最小高度占位，减少内容突变。

#### 阶段 C（2 周）：可观测与验收

1. 建会员端 UX 指标：
   - route_switch_duration_p95
   - blank_frame_duration
   - scroll_restore_success_rate
   - pull_to_refresh_abort_rate
2. 上线 A/B：新旧切页方案对比（闪跳投诉率、停留时长、转化率）。

### 12.4 你要的"完善会员前端体验逻辑"参考架构

- **布局层（常驻）**：Header/BottomNav/客服/滚动容器常驻不卸载。
- **页面层（可替换）**：只替换 `Outlet` 内容。
- **数据层（可中断）**：所有请求可取消，切页自动 abort。
- **刷新层（轻量）**：刷新数据而不是刷新页面。
- **观测层（可量化）**：每次路由切换都上报耗时与白屏片段。

### 12.5 上线前验收标准（会员体验）

1. 任意会员页面互跳，不出现整页白屏闪烁。
2. 切页后底栏不抖动，客服按钮位置稳定。
3. 返回上一页时滚动位置恢复成功率 ≥ 95%。
4. 弱网（Fast 3G）下首跳页面白屏时长显著下降。
5. 下拉刷新不会触发整页重载。

### 13.2 优先级 P1（重复/冗余逻辑，建议结构优化）

#### P1-1 抽奖前端服务层存在"薄封装重复"

发现：`MemberSpin` 在抽奖结果后既手动拉取 quota/logs，又做 query invalidation，存在冗余请求机会。

建议：

- 统一"单一真源刷新策略"：要么 invalidate + queryFn，要么局部 optimistic + 单次 refetch。

影响：

- 对低端机/弱网用户，可能加重初次会话流量与主线程压力。

建议：

- 改为"基于访问概率/空闲预算"的分级预加载（只预热高概率下一跳）。

结论：**可安全保留（短期），但建议后续优化**。
