# 部署诊断：本地与线上不一致

## 一、如何验证版本是否一致

1. **本地**：运行 `npm run build` 后，用 `npm run preview` 预览，点击右上角头像，查看下拉菜单中的「构建: YYYY-MM-DD HH:mm:ss」
2. **线上**：打开 https://crm.fastgc.cc，点击右上角头像，查看「构建」时间
3. **对比**：若本地构建时间与线上显示一致，说明部署成功；若不一致，按下方步骤排查

---

## 二、常见原因与处理

### 1. PWA / Service Worker 缓存（最常见）

**现象**：部署成功，但浏览器仍显示旧版本。

**处理**：
- **强制刷新**：`Ctrl + Shift + R`（Windows）或 `Cmd + Shift + R`（Mac）
- **清除站点数据**：开发者工具 → Application → Storage → Clear site data
- **关闭后重开**：关闭该站点所有标签页，重新打开

### 2. 自定义域名与部署分支不一致（crm.fastgc.cc 与 xxx.pages.dev 不同）

**现象**：`https://954c2399.gift-system.pages.dev` 正常，但 `https://crm.fastgc.cc` 显示旧版或异常。

**原因**：自定义域名 `crm.fastgc.cc` 绑定在 **main** 分支。若 `wrangler deploy` 未指定 `--branch=main`，会部署到默认分支（如 master），导致自定义域名仍指向旧部署。

**处理**：`deploy-full.ps1` 已改为使用 `--branch=main`，确保部署到 main 分支，使 crm.fastgc.cc 显示最新版本。

### 3. Cloudflare 同时启用 Git 构建与直接上传

**现象**：`npm run deploy:full` 执行成功，但线上被 Git 构建覆盖。

**原因**：若 Cloudflare Pages 已连接 GitHub，每次 push 会触发云端构建；`wrangler pages deploy` 是直接上传。若 Git 构建在直接上传之后完成，生产环境会切到 Git 构建版本。

**处理**：
1. 登录 [Cloudflare Dashboard](https://dash.cloudflare.com) → Pages → gift-system
2. 若存在「连接到 Git」：建议**关闭 Git 构建**，仅使用 `wrangler pages deploy` 直接上传
3. 或保留 Git 构建：在 Cloudflare 中配置与本地相同的环境变量（如 `VITE_SUPABASE_URL` 等），并确保每次发布都先 push 再执行 deploy

### 4. 部署了错误目录

**现象**：本地修改未体现在线上。

**确认**：`deploy-full.ps1` 中 `$projectRoot = Split-Path -Parent $PSScriptRoot`，即脚本所在目录的上一级。请确认执行 `npm run deploy:full` 时，当前项目为 `Cursor-huiyuanxitong-main\Cursor-huiyuanxitong-main`。

### 5. 环境变量差异

**现象**：功能或接口地址与本地不一致。

**确认**：构建时使用项目根目录的 `.env`。线上若使用 Cloudflare Git 构建，需在 Cloudflare 项目设置中配置相同变量。

## 三、推荐发布流程

```powershell
# 1. 确认在正确项目目录
cd "项目路径\Cursor-huiyuanxitong-main\Cursor-huiyuanxitong-main"

# 2. 确认无未提交修改
git status

# 3. 执行完整发布（Git 推送 + 构建 + Cloudflare 上传）
npm run deploy:full "fix: 描述本次修改"

# 4. 部署完成后，强制刷新浏览器验证
# Ctrl+Shift+R 打开 https://crm.fastgc.cc
# 点击头像查看「构建」时间
```
