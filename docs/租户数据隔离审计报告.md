# 租户数据隔离审计报告

> 审计日期：2026-03-10  
> 审计范围：orders、members、employees、RPC 函数、前端数据获取

---

## 一、严重问题（P0）

### 1.1 核心表 RLS 无租户过滤

| 表 | 当前 RLS 策略 | 问题 |
|---|---|---|
| **orders** | `orders_employee_select`: 仅校验 `has_role(admin/manager/staff)` | 任意租户的 admin/manager/staff 可查看**全部订单** |
| **members** | `members_employee_select`: 仅校验 `has_role(admin/manager/staff)` | 任意租户可查看**全部会员** |
| **employees** | `employees_all_authenticated_select`: 仅校验 `has_role(admin/manager/staff)` | 任意租户可查看**全部员工** |

**影响**：租户 A 的 002 用户登录后，可看到租户 B 的 003 订单、会员、员工，数据完全未隔离。

**来源**：`20260115140939_remix_migration_from_pg_dump.sql`、`20260130073802_283438b2-1dc1-4cef-889b-7ef7fa83f43b.sql`

---

### 1.2 get_order_filter_stats 移除租户过滤

| 文件 | 变更 |
|---|---|
| `run-fixes.sql` | 移除 `v_tenant_id` 及 `EXISTS (... tenant_id = v_tenant_id)` 过滤 |
| `20260309170000_fix_order_stats_platform_admin_all_orders.sql` | 同上，`p_tenant_id` 参数存在但未使用 |

**影响**：订单统计（卡值总和、利润总和、交易用户）返回**全库**数据，而非当前租户。

**前端**：`orderQueries.ts` L387 固定传 `p_tenant_id: null`，平台超管查看租户时也未传 `viewingTenantId`。

---

### 1.3 get_dashboard_trend_data 移除租户过滤

| 文件 | 变更 |
|---|---|
| `run-fixes.sql` | 移除 `v_tenant_id` 及 `COALESCE(e.tenant_id, e_creator.tenant_id) = v_tenant_id` 过滤 |

**影响**：仪表盘「交易用户、订单量、交易额、利润」等统计返回**全库**数据。

**对比**：`20260309160000_fix_trading_users_phone_fallback.sql` 仍保留租户过滤，但 `run-fixes.sql` 覆盖了该逻辑。

## 二、高优先级问题（P1）

### 2.1 profile 无 employee_id 时 v_tenant_id 为 NULL

- 当用户 `profiles.employee_id` 为空或 `employees` 表中无对应记录时，`v_tenant_id` 为 NULL。
- 原逻辑：`(v_tenant_id IS NULL OR ...)` 会放行所有订单。
- **修复**：无法解析租户时，应拒绝访问或返回空结果，而不是放宽为全库。

### 2.2 fetchOrderStats 未传 viewingTenantId

- `orderQueries.ts` L375–387：`fetchOrderStats` 始终传 `p_tenant_id: null`。
- 平台超管查看某租户时，应传入 `viewingTenantId` 作为 `p_tenant_id`。
- **修复**：调用方传入 `tenantId`，并传给 RPC。

---

## 三、中优先级问题（P2）

### 3.1 get_active_employees_safe / get_active_visible_employees_safe

- 无租户过滤，返回全库员工。
- **修复**：增加 `tenant_id` 过滤，或改为仅返回当前用户租户员工。

### 3.2 useEmployees.initializeCache 直接查全表

- `useEmployees.ts` 中 `initializeCache` 直接查 `employees` 表，无租户过滤。
- **修复**：按当前用户租户过滤，或通过 RPC 获取。

## 五、修复建议

### 5.1 数据库层（新建 migration）

1. **orders 表 RLS 增加租户过滤**：
   - 仅允许访问 `creator_id` 或 `sales_user_id` 属于当前用户租户的订单。
   - 平台超管需单独策略（`is_platform_super_admin` 可访问全部）。

2. **members 表 RLS 增加租户过滤**：
   - 仅允许访问 `creator_id` 或 `recorder` 属于当前用户租户的会员。
   - 平台超管需单独策略。

3. **employees 表 RLS 增加租户过滤**：
   - 仅允许访问 `tenant_id` 与当前用户租户相同的员工。
   - 平台超管需单独策略。

4. **恢复 get_order_filter_stats 租户过滤**：
   - 恢复 20260309120000 的 `v_tenant_id` 逻辑。
   - 当 `v_tenant_id` 为 NULL 且非平台超管时，返回空结果。

5. **恢复 get_dashboard_trend_data 租户过滤**：
   - 重新引入 `v_tenant_id` 及对应 WHERE 条件。
   - 对 `v_tenant_id` 为 NULL 的情况做安全处理。

### 5.2 前端 / 服务层

1. **fetchOrderStats**：平台超管查看租户时传入 `viewingTenantId` 作为 `p_tenant_id`。
2. **useMembers / useEmployees**：在非平台超管模式下，优先通过 RPC 获取数据，或确保 RLS 已正确限制租户后再使用直接表查询。

### 5.3 架构建议

- 租户隔离应以 **RLS 为主**，应用层和 RPC 为辅。
- 避免依赖「RLS 允许全库访问」的假设来设计统计函数。
- 对 profile 无 employee_id 的用户，应限制或禁止访问业务数据。
