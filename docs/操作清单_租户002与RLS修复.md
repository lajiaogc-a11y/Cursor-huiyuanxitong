# 操作清单：租户 002 数据可见性修复 + RLS 漏洞修复

纯文本操作清单，按时间顺序记录从 RLS 漏洞发现到租户 002 员工数据可见性修复的全部操作。

---

## 一、问题发现

### 1. RLS 漏洞
- **原因**：`20260311000002_restore_disable_rls_temporarily.sql` 临时禁用了多张表的 RLS，用于数据恢复，但恢复完成后未重新启用
- **影响表**：orders、members、employees、profiles、activity_gifts、points_ledger 等 40+ 张表
- **风险**：数据可被任意已认证用户直接读取/修改，存在安全漏洞

### 2. 租户 002 员工看不到数据
- **现象**：平台超管「进入租户 002」能看到订单和会员，但租户 002 员工直接登录后看不到任何数据
- **数据状态**：orders 表 808 条、members 表 550 条，数据未丢失

---

## 二、数据库迁移操作（按执行顺序）

### 1. RLS 重新启用（20260321000000_fix_rls_security_vulnerabilities.sql）
- 重新启用所有曾被 `20260311000002` 禁用的表的 RLS
- 涉及表：activity_gifts、activity_reward_tiers、activity_types、api_keys、audit_records、balance_change_logs、card_types、cards、currencies、customer_sources、data_settings、employee_login_logs、employee_name_history、employee_permissions、employees、exchange_rate_state、invitation_codes、knowledge_articles、knowledge_categories、knowledge_read_status、ledger_transactions、member_activity、members、operation_logs、orders、payment_providers、permission_change_logs、permission_versions、points_accounts、points_ledger、profiles、referral_relations、report_titles、role_permissions、shared_data_store、shift_handovers、shift_receivers、user_data_store、vendors、webhook_delivery_logs、webhooks
- 为 spins、prizes、redemptions、check_ins、member_invites、member_transactions、otp_verifications、tenants 等新表启用 RLS
- 为 tenants、spins、prizes、redemptions 等无策略表添加最小必要 RLS 策略

### 2. 租户 002 数据恢复与 RLS 调整（20260323000000 ~ 20260328000000）
- `20260323000000_restore_tenant_002_all_data.sql`：恢复租户 002 相关数据
- `20260324000000_fix_tenant_002_data_visibility.sql`：修复 platform_get_tenant_* RPC，使 002 孤儿数据可见
- `20260325000000_fix_tenant_002_employee_rls.sql`：RLS 增加 002 孤儿数据可见逻辑
- `20260326000000_allow_tenant_employee_platform_rpc.sql`：RPC 允许租户员工调用 platform_get_tenant_* 获取本租户数据
- `20260327000000_force_restore_tenant_002_all_data.sql`：强制员工归属 002
- `20260328000000_tenant_002_see_all_aggressive.sql`：002 员工可查看全部订单/会员，RPC 对 002 返回全部数据

### 3. 租户员工专用 RPC（20260329000000_tenant_employee_my_orders_rpc.sql）
- 创建 `get_my_tenant_orders_full()`：根据当前用户 employee.tenant_id 返回本租户订单（无参数）
- 创建 `get_my_tenant_usdt_orders_full()`：同上，USDT 订单
- 创建 `get_my_tenant_members_full()`：同上，会员
- 创建 `get_my_tenant_dashboard_trend()`：同上，仪表盘趋势
- 兜底逻辑：当 profiles.employee_id 为空时，用 profile.email 前缀匹配 employees.username 获取 tenant_id

### 4. verify 函数修复（20260330000000_fix_tenant_employee_data_visibility.sql）
- DROP 并重建 `verify_employee_login_detailed`
- 增加 `tenant_id` 列到返回值，供登录后立即设置 employee.tenant_id

## 三、前端改动操作

### 1. TenantViewContext.tsx
- 移除对 `supabase.from("tenants")` 的查询（避免 RLS 拦截）
- 租户员工直接设置 `viewingTenantId = employee.tenant_id`，不依赖 tenants 表
- 移除 supabase 导入

### 2. tenantService.ts
- 新增 `getMyTenantOrdersFull()`：调用 `get_my_tenant_orders_full`
- 新增 `getMyTenantUsdtOrdersFull()`：调用 `get_my_tenant_usdt_orders_full`
- 新增 `getMyTenantMembersFull()`：调用 `get_my_tenant_members_full`
- 新增 `getMyTenantDashboardTrend()`：调用 `get_my_tenant_dashboard_trend`

### 3. orderQueries.ts
- `fetchOrdersFromDb` 增加参数 `useMyTenantRpc`
- `fetchUsdtOrdersFromDb` 增加参数 `useMyTenantRpc`
- `fetchOrderStats` 增加参数 `useMyTenantRpc`
- 当 `useMyTenantRpc` 为 true 时，调用 `getMyTenant*` 而非 `getTenant*`

### 4. useOrderList.ts
- 计算 `useMyTenantRpc = !!(effectiveTenantId && employee?.tenant_id && effectiveTenantId === employee.tenant_id)`
- 将 `useMyTenantRpc` 传入 `fetchOrdersFromDb`

### 5. useUsdtOrderList.ts
- 同上，计算 `useMyTenantRpc` 并传入 `fetchUsdtOrdersFromDb`

### 6. useOrderStats.ts
- 同上，计算 `useMyTenantRpc` 并传入 `fetchOrderStats`

### 7. useMembers.ts
- `fetchMembersFromDb` 增加参数 `useMyTenantRpc`
- 计算 `useMyTenantRpc` 并传入，当为 true 时调用 `getMyTenantMembersFull`

### 8. useDashboardTrend.ts
- `fetchTrendData` 增加参数 `useMyTenantRpc`
- 计算 `useMyTenantRpc` 并传入，当为 true 时调用 `getMyTenantDashboardTrend`

### 9. AuthContext.tsx
- `verify_employee_login_detailed` 返回结果中读取 `result.tenant_id`
- 构建 emp 对象时加入 `tenant_id: result.tenant_id ?? null`
- `employeeInfo` 中加入 `tenant_id: emp.tenant_id ?? null`，使登录后立即有 effectiveTenantId
